# Member 1 Feature Engineering Pipeline

## Architecture Overview
The Feature Engineering pipeline (Member 1) is responsible for taking raw candidate profiles and extracting structured, normalized features (scores ranging from 0.0 to 1.0). In addition to feature extraction, the pipeline incorporates trap detectors to identify suspicious candidate behaviors such as keyword stuffing and skill inflation.

The pipeline comprises two main types of modules:
1. **Preprocessing Modules**: Analyze various aspects of the candidate profile (skills, career history, title, behavior, experience) and generate normalized scores.
2. **Trap Detectors**: Identify "honeypot" conditions or unnatural patterns in resumes to flag potentially fraudulent or heavily exaggerated profiles.

## Feature Descriptions
- **`career_features.py`**: Extracts the `career_score` by evaluating the candidate's profile summary and career history against specific, weighted AI/ML categories (e.g., Retrieval Systems, Vector Databases, RAG). Rewards candidates with evidence in distinct categories rather than just counting keywords.
- **`title_features.py`**: Generates a `title_score` by evaluating the relevance of the candidate's current and past job titles against AI/ML engineering roles.
- **`skill_features.py`**: Calculates a `skill_score` based on the candidate's proficiency and duration with key technical skills.
- **`behavioral_features.py`**: Derives a `behavior_score` by analyzing behavioral indicators or soft skills.
- **`experience_features.py`**: Computes an `experience_score` based on total years of experience, normalizing it to a 0.0-1.0 scale.
- **`feature_extractor.py`**: Orchestrates the execution of the individual feature modules and aggregates all scores into a unified feature vector.

## Trap Descriptions
- **`keyword_stuffer.py`**: Identifies candidates who list an excessive number of AI skills (e.g., 5-10+) but have little to no supporting evidence in their career history (`career_score`). High discrepancy results in a flag and a proportional penalty.
- **`honeypot_detector.py`**: Flags candidates with unrealistic profiles. Detection criteria include:
  - Skill inflation (too many "expert" skills).
  - Unrealistic expertise (claiming "expert" proficiency with very low duration, e.g., < 1 year).
  - Title inflation (Senior/Lead/Director titles with very low years of experience).
  - Impossible skill timelines (e.g., skill durations exceeding total work experience).

## Output Schema
The pipeline produces a processed dataset (`candidate_features.csv`) with the following general schema:

| Column Name | Type | Description |
| :--- | :--- | :--- |
| `candidate_id` | String | Unique identifier for the candidate |
| `title_score` | Float [0.0 - 1.0] | Normalized score for job titles |
| `career_score` | Float [0.0 - 1.0] | Normalized score for career history |
| `skill_score` | Float [0.0 - 1.0] | Normalized score for skills |
| `behavior_score` | Float [0.0 - 1.0] | Normalized score for behavior |
| `experience_score` | Float [0.0 - 1.0] | Normalized score for experience |
| `keyword_stuffer_flag` | Boolean | True if candidate is flagged for keyword stuffing |
| `keyword_stuffer_penalty` | Float [0.0 - 1.0] | Penalty proportional to confidence of keyword stuffing |
| `honeypot_flag` | Boolean | True if candidate is flagged by honeypot detector |
| `honeypot_penalty` | Float [0.0 - 1.0] | Penalty based on the severity of the honeypot detection |
| `honeypot_confidence` | Float [0.0 - 1.0] | Confidence level of the honeypot detection |

*(Note: Exact column names may vary slightly based on the final pandas dataframe generation in `feature_extractor.py`.)*

## Integration Instructions for Member 2
Member 2 (Model Training / Ranking) should use the output of this pipeline as input for the ranking or classification models. 

1. **Input Data**: Read the `candidate_features.csv` file generated by Member 1's pipeline.
2. **Handling Flags and Penalties**:
   - The trap detectors provide both a `flag` and a `penalty` (and `confidence`). 
   - You can choose to outright filter out candidates where a flag is `True` (hard filtering).
   - Alternatively, you can incorporate the `penalty` into the final ranking formula (soft filtering). For example: `final_score = raw_model_score * (1.0 - keyword_stuffer_penalty - honeypot_penalty)`.
3. **Feature Vectors**: The scores (`career_score`, `skill_score`, etc.) are pre-normalized to [0.0, 1.0], making them directly suitable as features for machine learning models (e.g., Logistic Regression, Gradient Boosting, or Neural Networks).
4. **Validation**: Ensure that candidates with high trap penalties are closely examined or automatically down-ranked in the final evaluation to maintain the quality of the top candidates.
